Microblog

Background

Microblog is a blogging app built with React and Redux

Key Learnings

‣ Using redux-thunk to handle asynchronous actions
‣ The benefits of normalizing state shape (to simplify reducer logic, prevent unnecessary re-renders, and improve performance)

Challenges

One of the interesting challenges in this project was thinking about the best shape for our state. For each post we wanted to keep track of the post title, description, content, and comments; and for each comment we wanted to keep track of the content. We would also need to support the ability to edit & delete both posts and comments.

An initial thought was to structure our state as follows: 

state = {
	posts = {
		id: {
			title: ,
			description: ,
			body: ,
			comments: [ {id, text}, ...]
			}
		}
	};

The structure of our state started becoming pretty nested though, which also increased the complexity of our reducer logic. Furthemore, since reducers must be pure functions (meaning they do not mutate data), we need to copy all ancestors in the state tree; and new object references will cause components to re-render, often unncessarily.

I had a feeling there was a better shape for our state, so after the project, I did some research on best practices and learned about normalizing state shape.

Further Work

To simplify reducer logic and improve performance (by eliminating unnecessary re-rendering), I would consider normalizing our state shape 1 to look like the following: 

state = {
	posts = {
		byId: {
			"post1": {
				id: "post1",
				author: "user1",
				body: ".....",
				comments: ["comment1", ...]
			},
		},
		allIds: ["post1", ...]
	},

	comments: {
		byId: { ... },
		allIds: [ ... ]
	},

	authors: {
		byId: { ... },
		allIds: [ ... ]
	}
};

This flatter state structure offers several improvements:

‣ The reducer logic will be simpler because there is less nesting
‣ Since each data type is separated, updating one portion of the tree (like a specific comment) will only require an update to that part of the tree, which means fewer portions of the UI (like the post) will update because its data will be unchanged.
‣ Because each item is only defined in one place, we don't have to worry about making changes in multiple places if that item is updated.

Footnotes

1 The basic concepts of normalizing data are as follows:
• Each type of data should get its own "slice" in the state
• Each "slice" should store individual items in an object, with IDs as keys and item themselves as values
• Any references to individual items should be done by ID
• Arrays of IDs should be used to indicate ordering
